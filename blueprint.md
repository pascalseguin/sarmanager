# SAR Manager — Blueprint

**Stack:** Electron 28 desktop app · Express backend (port 3737) · React + Vite frontend · SQLite via better-sqlite3  
**Roles:** `sm` (Search Manager — full access) · `searcher` (Ground Searcher — auth + weather only)  
**Previous version:** [blueprint-archived.md](blueprint-archived.md)

---

## 3. Key Workflows

### Publish to CalTopo
1. `POST /api/caltopo/publish/:operationId`
2. If no `caltopo_map_id`: `provisionMap` → looks up "SEASAR Operations" collection (5s timeout) → `createMap` with parsed title `location-yyyy-mm-dd-D4H0000000`
3. `publishFeatures` → IPP marker with `icp` icon, secondary PLS/LKP marker, 50/75/95% probability rings
4. Auth failure on existing map → re-provision fresh map and retry once

### Task Lifecycle
1. SM creates task: `POST /api/tasks` → status `assignment_prepared`
2. SM assigns team: `POST /api/tasks/:id/assign` → personnel status → `deployed`, task → `in_field`
3. Team returns: `PUT /api/tasks/:id/debrief` → personnel status → `available`, task → `debriefed`

### D4H Callout Flow
1. `POST /api/d4h/callout` → creates D4H incident, stores callout record
2. Poll `GET /api/d4h/callout/:calloutId/responses` → syncs attendance, updates local personnel status

### UTM Coordinates
- All coordinates stored as WGS84 decimal degrees in SQLite
- `convertToUTM()` runs at API response time (not stored)
- All operation GET responses include `lkp_utm`, `pls_utm`, `active_ipp_utm`
- `active_ipp_utm` reflects whichever point (`lkp` or `pls`) is set as `ipp_type`
